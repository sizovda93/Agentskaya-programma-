# Промт для Replit: Мобильное приложение «Агентум Про»

## Описание проекта

Создай **мобильное веб-приложение** (PWA) — мобильный клиент для существующей платформы **«Агентум Про»** (agentum.club). Это партнёрская SaaS-платформа юридической компании по банкротству физических лиц (БФЛ).

### ВАЖНО: Синхронизация с платформой
Приложение подключается к **той же самой PostgreSQL базе данных**, что и основная платформа. Это значит:
- **НЕ нужно создавать миграции** — все таблицы уже существуют
- **Данные синхронизированы в реальном времени** — то, что создаёт партнёр в мобильном приложении, мгновенно видно менеджеру на платформе, и наоборот
- Аутентификация проверяет тех же пользователей из таблицы `profiles`
- Все лиды, сообщения, выплаты, прогресс обучения — общие

### Целевая аудитория мобильного приложения
Приложение предназначено **только для роли «Партнёр» (agent)**. Менеджеры и админы работают через веб-платформу. Партнёр использует мобильное приложение для:
- Передачи клиентов (создание лидов)
- Отслеживания статусов лидов и выплат
- Общения с менеджером
- Прохождения обучения
- Просмотра объявлений и вакансий

## Стек технологий

- **Framework:** Next.js (App Router), React, TypeScript
- **UI:** Tailwind CSS, Radix UI (shadcn/ui), Lucide Icons
- **Backend:** Next.js API Routes — подключение к **существующей БД** через `pg` (raw SQL)
- **Аутентификация:** JWT (jsonwebtoken), хранение токена в localStorage (для мобильного)
- **PWA:** Manifest + Service Worker для установки на телефон
- **Шрифт:** Inter (кириллица)
- **Тема:** Тёмная (по умолчанию) + светлая

## Аутентификация

### Особенности для мобильного приложения
В отличие от основной платформы (httpOnly cookies), мобильное приложение хранит JWT в **localStorage**:

```typescript
// Вход: POST /api/auth/login
// Body: { email, password }
// Response: { token, user: { id, role, fullName, email } }

// Клиент сохраняет токен:
localStorage.setItem('token', data.token);

// Все запросы отправляют токен в заголовке:
headers: { 'Authorization': `Bearer ${token}` }
```

### JWT совместимость
Используй **тот же JWT_SECRET**, что и основная платформа. Это позволяет:
- Токены, выданные мобильным приложением, валидны на платформе (и наоборот)
- Payload: `{ userId: string, role: 'agent' | 'manager' | 'admin' }`
- Срок жизни: 7 дней

### API middleware (для мобильного)
```typescript
// Каждый API route проверяет токен из Authorization header:
import jwt from 'jsonwebtoken';

export function getSessionFromHeader(req: Request) {
  const auth = req.headers.get('Authorization');
  if (!auth?.startsWith('Bearer ')) return null;
  
  try {
    const payload = jwt.verify(auth.slice(7), process.env.JWT_SECRET!) as { userId: string; role: string };
    return payload;
  } catch {
    return null;
  }
}
```

### Вход
- Email + пароль
- Проверка: `SELECT id, role, full_name, email, password_hash FROM profiles WHERE email = $1`
- Сравнение: `bcrypt.compare(password, password_hash)`
- **Допускать только role='agent'** (менеджеры и админы работают через веб)
- Кнопка "показать/скрыть пароль" (иконка глаза)
- При успехе — сохранить токен, редирект на главную

### Забыли пароль (сброс через Telegram)
Флоу из 3 шагов на одном экране `/forgot-password`:

**Шаг 1 — Ввод email:**
- Пользователь вводит email
- API `POST /api/auth/forgot-password` с `{ email }`
- Сервер ищет пользователя в `profiles`, проверяет привязку Telegram в `telegram_bindings`
- Если Telegram привязан → генерирует 6-значный код, сохраняет в `password_reset_tokens` (срок 15 мин), отправляет код в Telegram через бота
- Если Telegram НЕ привязан → показать сообщение: «Telegram не привязан. Обратитесь к менеджеру для сброса пароля.»
- Не раскрывать, существует ли email (безопасность)

**Шаг 2 — Ввод кода + новый пароль:**
- Поле для 6-значного кода (только цифры, крупный шрифт, tracking-widest)
- Поле для нового пароля (мин. 6 символов, кнопка показать/скрыть)
- Кнопка «Отправить код повторно» → возврат к шагу 1
- API `POST /api/auth/reset-password` с `{ email, code, newPassword }`
- Сервер проверяет код в `password_reset_tokens` (не использован, не истёк)
- Хеширует новый пароль через bcrypt, обновляет `profiles.password_hash`

**Шаг 3 — Готово:**
- Зелёная галочка, текст «Пароль успешно изменён!»
- Кнопка «Войти с новым паролем» → редирект на `/login`

**Ссылка на экране входа:**
- Между кнопкой «Войти» и «Нет аккаунта? Регистрация» добавить ссылку «Забыли пароль?»

### Регистрация
- Поля: ФИО, email, телефон, пароль, город
- Создание записи в `profiles` (role='agent') + `agents`
- Генерация `ref_code` (8 случайных символов)
- Присвоение `partner_number` (SELECT MAX(partner_number) + 1 FROM agents, минимум 888)
- Поддержка `?ref=CODE` — сохраняется для реферальной статистики

  - Зелёный бар: «Партнёру Иванов И.И. выплачено 15 000 ₽»
  - Синий бар: «Списано 1 500 000 ₽ долгов — Петров П.П. (12.03.2026)»
- **Быстрые действия** (2 кнопки):
  - «Передать клиента» → экран создания лида

- **Прогресс обучения** — прогресс-бар по обязательным модулям

### 2. Лиды
**Путь:** `/leads`

- **Список лидов** партнёра (pull-to-refresh):
  ```sql
  SELECT l.*, a.id as agent_id FROM leads l
  JOIN agents a ON l.assigned_agent_id = a.id
  WHERE a.user_id = $1
  ORDER BY l.created_at DESC
  ```
- Каждый лид — карточка: ФИО, телефон, статус (цветной бадж), дата
- **Фильтр по статусу** (горизонтальные чипсы): Все / Новый / В работе / Выигран / Потерян
- **Кнопка «+» (FAB)** → форма создания лида
- **Детали лида** (tap → `/leads/[id]`):
  - Полная карточка: ФИО, телефон, email, город, комментарий, сумма долга
  - Статус с таймлайном событий
  - Нельзя редактировать статус (только менеджер)

#### Создание лида
- Поля: ФИО*, Телефон*, Email, Город, Комментарий, Оценка суммы долга

### 3. Финансы
**Путь:** `/finance`

- **Ожидает выплаты**: сумма pending + processing
- **Список выплат** (карточки):
  ```sql
  SELECT p.* FROM payouts p
  JOIN agents a ON p.agent_id = a.id
  ```

- Каждая выплата: сумма, статус (pending→жёлтый, processing→синий, paid→зелёный, rejected→красный), имя клиента, дата

## КЛЮЧЕВОЕ ОТЛИЧИЕ: Синхронизация с основной платформой

Это приложение — **НЕ отдельный продукт**, а мобильный клиент для уже работающей платформы **agentum.club**.

### Что это значит на практике:

1. **Общая база данных** — приложение подключается к **той же самой PostgreSQL**, что и основная платформа. Все данные (лиды, сообщения, выплаты, прогресс обучения) синхронизированы в реальном времени без каких-либо дополнительных механизмов.

2. **НЕ создавай миграции, таблицы, ENUM** — вся схема уже развёрнута и работает. Просто подключайся и делай SELECT/INSERT/UPDATE по существующим таблицам.

3. **Совместимые JWT** — используется тот же `JWT_SECRET`. Если партнёр залогинился через мобильное приложение, его токен валиден и на платформе (и наоборот). Не нужен отдельный auth-сервер.

4. **Двусторонняя видимость:**
   - Партнёр создаёт лида в приложении → менеджер мгновенно видит его на платформе
   - Менеджер пишет сообщение на платформе → партнёр видит его в приложении при следующем polling (каждые 5 сек)
   - Менеджер меняет статус лида → партнёр видит обновлённый статус
   - Менеджер создаёт выплату → партнёр видит её в разделе «Финансы»
   - Админ публикует объявление → оно появляется в приложении

5. **Нет дублирования бизнес-логики** — всё, что создаёт менеджер или админ (объявления, вакансии, тикеры, модули обучения, договоры, маркетинговые материалы), автоматически доступно в мобильном приложении, потому что данные лежат в одних и тех же таблицах.

6. **Один пользователь — один аккаунт** — партнёр регистрируется один раз (неважно, через веб или приложение). Его профиль, прогресс обучения, рефералы — всё едино.

```bash
# 4. Перезапустить PostgreSQL
```

### Категория: howto (Инструкции)

**1. Где размещать посты** (featured)
Описание: Подробная инструкция по площадкам

```
ВКонтакте
Разместите пост на своей стене. Если вы состоите в местных городских группах — напишите пост там (если правила позволяют). Также можно оставлять полезные комментарии под постами о финансах и долгах.

Telegram
Если у вас есть свой канал — публикуйте посты регулярно (1-2 раза в неделю). Также можно предложить пост в тематические каналы вашего города.

Одноклассники
Хорошая площадка для аудитории 35+. Разместите пост на своей странице и в группах.

Instagram
Используйте сторис — они показываются подписчикам в первую очередь. В описании профиля разместите реферальную ссылку.

Важно
Не спамьте. Лучше один качественный пост в неделю, чем десять одинаковых за день. Отвечайте на комментарии и вопросы — это повышает доверие.
```

**2. Как писать в личные сообщения**
Описание: Правила хорошего тона при личной рассылке

```
Главное правило
Пишите только тем, кому это реально может быть полезно. Массовая рассылка незнакомым людям — это спам, и аккаунт могут заблокировать.

Кому писать:
• Знакомым, которые упоминали проблемы с долгами
• Друзьям, у которых вы знаете о финансовых трудностях
• В ответ на посты/сторис, где люди жалуются на долги

Как писать:
1. Начните с приветствия — по имени
2. Объясните контекст — почему пишете именно ему
3. Не давите — предложите «просто посмотреть»
4. Дайте ссылку на консультацию
5. Закончите фразой «без обязательств»

Повторное касание
Если человек не ответил — подождите 5-7 дней. Напишите коротко: «Привет, напоминаю о возможности». Если снова тишина — не пишите больше.

Чего НЕ делать:
• Не обещайте 100% списание долгов
• Не пишите капсом и не используйте «СРОЧНО!!!»
• Не добавляйте в группы без спроса
• Не рассылайте одинаковый текст десяткам людей подряд
```
